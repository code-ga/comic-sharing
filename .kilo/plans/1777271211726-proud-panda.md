# Plan: API Proxy to Hide Backend URL

## Goal
Hide the direct backend URL by routing all API requests through the frontend. Client requests to `/api/*` will be forwarded to the actual backend, with the backend URL configurable via environment variables (server-side only).

## Approach
- Use **Next.js App Router API Route** as a reverse proxy catch-all.
- The frontend Eden client will call `/api/...` (relative path), never exposing the real backend host.
- The proxy route runs on the Next.js server, forwards requests to the target backend defined in `BACKEND_TARGET_URL`, and returns the response.

## Why API Route over rewrites
- Full control over request/response headers and body.
- Ability to add logging, error handling, and transformations later.
- Works consistently across all Next.js versions (including the project's custom build).
- Simple, self-contained implementation with no extra dependencies.

## Technical Changes

### 1. Update `frontend/constants.ts`
- Change `BACKEND_URL` from absolute URL to the local proxy base: `'/api/'`.
- Remove exposure of `VITE_BACKEND_URL` to client bundle; backend URL is server-side only.
- Result:
  ```ts
  export const BACKEND_URL = '/api/';
  ```

### 2. Create Proxy Route: `frontend/app/api/[...path]/route.ts`
- Catch-all route that handles all HTTP methods (GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS).
- Implementation:
  - Extract the remaining path after `/api` and combine with query string.
  - Build target URL: `{BACKEND_TARGET_URL}{proxyPath}{search}`.
  - Clone incoming headers, excluding hop-by-hop headers (`host`, `connection`, `keep-alive`, etc.).
  - Forward request body (ReadableStream) and method.
  - Perform server-side `fetch` to the backend.
  - Return a `NextResponse` with status, body, and all headers from the backend (including `set-cookie`).
  - Basic error handling: on fetch failure, return 502 Bad Gateway.

- Important: Use Node.js runtime for reliable `process.env` and fetch behavior.
  ```ts
  export const runtime = 'nodejs';
  ```

### 3. Environment Configuration
- Add `BACKEND_TARGET_URL` as a **server-side** environment variable (no `NEXT_PUBLIC_` prefix).
- Update `.env.example` (or create `frontend/.env.example`) with:
  ```
  BACKEND_TARGET_URL=http://localhost:3001
  ```
- Update `docker-compose.yaml` for the `frontend` service:
  ```yaml
  environment:
    BACKEND_TARGET_URL: http://backend:3001
  ```
  (Remove or ignore any `BACKEND_URL` previously passed to frontend container; not needed.)

### 4. Backend CORS Considerations
- Current backend has CORS enabled with credentials.
- Since all client calls now hit the same origin (`/api`), CORS is not needed for those calls.
- Direct access to the backend (e.g., via Postman) may still require CORS; this is unaffected.
- Optionally, backend CORS could be restricted to the frontend origin in production, but not required for proxy to function.

## Implementation Steps
1. Modify `frontend/constants.ts` to set `BACKEND_URL = '/api/'`.
2. Create `frontend/app/api/[...path]/route.ts` with the proxy handler.
3. Add `BACKEND_TARGET_URL` to `.env.example`.
4. Update `docker-compose.yaml` to pass `BACKEND_TARGET_URL` to the frontend container.
5. Test locally:
   - Start backend (`bun dev` on :3001).
   - Start frontend (`bun dev` on :3000) with `BACKEND_TARGET_URL=http://localhost:3001`.
   - Verify API calls (e.g., `GET /api/permissions`) succeed.
6. Commit changes after validation.

## Validation
- Eden-generated API calls should resolve to `/api/...` and be intercepted by the Next.js route.
- Response data, status codes, and headers (especially cookies) should pass through unchanged.
- Network tab in browser should show same-origin requests to `localhost:3000/api/...`, not the backend host.

## Risks & Mitigations
| Risk | Mitigation |
|------|------------|
| Large request bodies may need streaming support | Use `req.body` directly; Node fetch streams automatically. |
| Backend redirects might break | Consider `redirect: 'manual'` in fetch and forward location header. For now, assume no redirects in API. |
| Hop-by-hop header leakage | Explicitly filter restricted headers when forwarding to backend. |
| Backend URL missing in env | Provide clear error in proxy if `BACKEND_TARGET_URL` is not set, logging the issue server-side. |

## Future Improvements
- Add request/response logging in the proxy for debugging.
- Implement rate limiting or authentication at the proxy layer.
- Support WebSocket upgrades if needed (unlikely for REST API).
- Add health check endpoint on frontend to verify proxy connectivity to backend.
