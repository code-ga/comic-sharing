# Fix TypeScript "never" errors in route handlers

## Problem
TypeScript errors across 4 route files with messages like:
- `Property 'id' does not exist on type 'never'` for `ctx.profile.id`, `ctx.user.id`, `ctx.profile?.id`
- `Property 'username' does not exist on type 'never'` for `ctx.body.username`
- `Property 'profileId'/'userId'/'type' does not exist on type 'never'` for `ctx.query`
- `Property 'id' does not exist on type 'never'` for `ctx.params.id`

**Affected files:**
- `src/routes/chapterImage.ts` (5 errors)
- `src/routes/chapters.ts` (4 errors)
- `src/routes/comic.ts` (3 errors)
- `src/routes/profile.ts` (14 errors)

## Root Cause Analysis

All affected route handlers use guards from `authenticationMiddleware` in `src/middleware/auth.ts`:
- `userAuth: true` - requires authentication, adds `ctx.user`, `ctx.session`, `ctx.profile`
- `roleAuth: "..."` - requires role, adds those plus `ctx.userPermissions`
- `optionalAuth: true` - optional auth, adds those properties optionally

The guard macros (`userAuth`, `roleAuth`, `optionalAuth`) are defined with `.macro({ ... })` but **lack a `schema` property**. In Elysia, the `schema` property in a macro is essential to inform TypeScript about what properties are injected into the context. Without it, TypeScript infers those context properties as `never` in strict mode.

When `ctx.profile`, `ctx.user`, `ctx.session`, `ctx.body`, `ctx.query`, or `ctx.params` are typed as `never`, all property accesses fail.

## Solution

Add proper `schema` definitions to each guard macro in `src/middleware/auth.ts` using Typebox (`@sinclair/typebox`) to describe the shape of injected values.

### Guard: `userAuth`
Adds (when authenticated):
- `user`: Better-Auth user object (minimally `{ id, email, name? }`)
- `session`: Better-Auth session object (minimally `{ id, expiresAt? }`)
- `profile`: Application profile from DB (`dbSchemaTypes.profile`)

### Guard: `roleAuth`
Same as `userAuth` plus:
- `userPermissions`: Array of permission strings (`Permission[]`), runtime is `Set<Permission>` but schema can be `Array(String)` for typing.

### Guard: `optionalAuth`
Adds properties optionally (may be absent or null):
- `user?: User | undefined`
- `session?: Session | undefined`
- `profile?: Profile | null` (can be `null` if session exists but no profile, or absent if no session)

## Implementation Steps

1. **Import necessary types** in `src/middleware/auth.ts`:
   - `Type` from `@sinclair/typebox`
   - `dbSchemaTypes` from `../database/types`
   - Optionally better-auth types if available, but inline minimal schema is sufficient.

2. **Define helper schemas** inline or as constants:
   - `userSchema`: object with `id: Type.String()`, `email: Type.String()`, `name: Type.Optional(Type.String())`
   - `sessionSchema`: object with `id: Type.String()`, `expiresAt: Type.Optional(Type.Date())` (optional)

3. **Update `userAuth` macro**:
   Add `schema: { user: userSchema, session: sessionSchema, profile: Type.Object(dbSchemaTypes.profile) }`

4. **Update `roleAuth` macro**:
   Add `schema: { user: userSchema, session: sessionSchema, profile: Type.Object(dbSchemaTypes.profile), userPermissions: Type.Array(Type.String()) }`

5. **Update `optionalAuth` macro**:
   Add `schema: { user: Type.Optional(userSchema), session: Type.Optional(sessionSchema), profile: Type.Optional(Type.Nullable(Type.Object(dbSchemaTypes.profile))) }`
   (nullable because they set `profile: profile || null`)

## Validation

After changes, run:
```bash
bun run typecheck
```
All previously reported errors (26 errors) should be resolved. No runtime behavior changes.

## Notes

- The `userSchema` and `sessionSchema` are minimal; only fields accessed in code (`user.id`) are required. Additional fields in Better-Auth's actual objects are okay because TypeScript structural typing allows extra properties.
- No changes needed to route files; the fix is entirely in the middleware guard definitions.
- Ensure imports do not create circular dependency: `auth.ts` already imports `db` from `../database`. Importing `dbSchemaTypes` from `../database/types` is safe because that file imports only schema definitions, not the db instance.

## Future Improvements

- Consider extracting schema definitions to a shared types file if reused elsewhere.
- Optionally use Better-Auth's exported Typebox schemas if available for more accurate typing.
