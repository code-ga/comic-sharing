## Current State Analysis

### Frontend Auth
- **`frontend/hooks/useAuth.ts`** - React Query-based auth hook providing:
  - `session` - Current session data (via `useQuery`)
  - `signIn` - Email/password sign in mutation
  - `signUp` - Email/password sign up mutation  
  - `createProfile` - Post-signup profile creation (via treaty API client)
  - `signOut` - Session invalidation and query cleanup
  - All mutations use `authClient` from `better-auth/react`

- **`frontend/lib/auth.ts`** - Auth client setup:
  - Uses `createAuthClient` from `better-auth/react`
  - Base URL: `/api/auth` (proxied through Next.js API routes)

### Backend Auth (Better-Auth)
- **`backend/src/libs/auth.config.ts`** - Better-Auth configuration:
  - Database adapter: Drizzle (PostgreSQL)
  - Email/password auth enabled with auto sign-in
  - Social providers: Google, GitHub, Discord (configured via env)
  - Cookie options: SameSite/lax (dev) or none (prod), secure in prod
  - Cross-subdomain cookies enabled
  - OpenAPI plugin enabled for API documentation
  - Tables: `user`, `session`, `account`, `verification`

- **`backend/src/routes/auth.ts`** - Auth route registration:
  - Mounts Better-Auth handler at `/api/auth` prefix
  - Minimal wrapper - auth is mostly self-contained

### Database Schema (Auth)
- **`backend/src/database/schema/auth.ts`** - Auth tables:
  - `user` - Core user (id, name, email, emailVerified, image, timestamps)
  - `session` - Session tokens (expiresAt, token, userId, ipAddress, userAgent)
  - `account` - Social/email provider accounts (OAuth, password)
  - `verification` - Email verification tokens
  - `role` - RBAC roles (permissions array, profileIds array)
  - `profile` - Extended user data (username, rolesIDs, userId foreign key)
  
  **Key Design Notes:**
  - Separation: `user` (auth-only) vs `profile` (business data)
  - Profile has `rolesIDs` (text array) for RBAC
  - Role has `adminRole` flag for superuser permissions
  - Role has `profileIds` array for many-to-many with profiles

### RBAC & Permissions System
- **`backend/src/constants/permissions.ts`**:
  - Resource-based permissions: `resource:action` format
  - Current resources: `user`, `role`
  - Filter DSL: `and`, `or`, `not` combinators
  - `resolveUserPermissions(roleIds)` aggregates permissions from roles
  - `adminRole` grants all permissions automatically

### Middleware & Authorization
- **`backend/src/middleware/auth.ts`** - Three auth strategies:
  1. **`userAuth`** - Requires valid session, optionally requires profile
     - Returns: `{ user, session, profile }`
  2. **`roleAuth`** - Requires session + profile + permission check
     - Returns: `{ user, session, profile, userPermissions }`
     - Evaluates `PermissionFilter` against user's permissions
  3. **`optionalAuth`** - Session optional, returns auth data if exists

### Profile Management
- **`backend/src/routes/profile.ts`** - Profile CRUD + auth integration:
  - Protected by `authenticationMiddleware` + `userAuth` guard
  - `GET /profile/me` - Current user's profile (requires auth)
  - `GET /profile/my-permissions` - Current user's permission list
  - `POST /profile/` - Create profile (first user = admin flag)
  - `PUT /profile/` - Update profile
  - `GET /profile/` - Get by profileId or userId
  - `GET /profile/list-user` - List all profiles (user:read required)
  - `GET /profile/search_user` - Search profiles (user:read required)
  - Uses `getInitialRoleIds()` to assign roles on creation

### Frontend Integration

#### Auth Client (Better-Auth)
- **`frontend/lib/auth.ts`** - Auth client setup:
  - Uses `createAuthClient` from `better-auth/react`
  - Base URL: `/api/auth` (proxied through Next.js API routes)
  - Used for: session management, sign in, sign out (Better-Auth built-ins)

#### Treaty API Client (Custom Resources)
- **`frontend/lib/api.ts`** - Treaty client for custom backend resources:
  - Uses `treaty` from `@elysiajs/eden` for type-safe API calls
  - Base URL: `BACKEND_URL = '/api/'` (proxied)
  - Used for: profile, comic, chapter, role (custom app resources)
  - Includes `getEdenErrorMessage()` utility
  - Provides typed access to backend via shared TypeBox schemas

**Key Distinction:**
- Better-Auth (`auth.ts`): Handles built-in auth (user, session, account, verification)
- Treaty API (`api.ts`): Handles custom app resources (profile, comics, chapters, roles)

#### Auth Hook
- **`frontend/hooks/useAuth.ts`** - Hook consumers use:
  - `sessionQuery` - Fetched via `authClient.getSession()` (Better-Auth)
  - `signIn`, `signUp`, `signOut` - Use `authClient` (Better-Auth)
  - `createProfile` - Uses `api` treaty client (custom resource endpoint)
  - Mutations invalidate/clear session queries on success
  - Error handling via `getEdenErrorMessage()` utility

## Data Flow

### Sign Up Flow
1. **Frontend**: `signUpMutation` → `authClient.signUp.email(email, password)` (Better-Auth)
2. **Better-Auth**: Validates input, creates `user` + `account` records in DB
3. **Better-Auth**: Creates session, returns session cookie to client
4. **Frontend**: Session query invalidated → triggers refetch
5. **Frontend**: `createProfileMutation` → `api.api.profile.post({ username })` (Treaty API)
6. **Backend**: Profile route → Creates `profile` record, assigns roles via `getInitialRoleIds()`
   - First user: Gets admin role if `createNewAdmin` app state is true

### Sign In Flow
1. **Frontend**: `signInMutation` → `authClient.signIn.email(email, password)` (Better-Auth)
2. **Better-Auth**: Verifies credentials against `user` + `account` tables
3. **Better-Auth**: Creates/updates session, returns session cookie
4. **Frontend**: Session query invalidated, triggers refetch
5. **Frontend**: Can fetch profile via `api.api.profile.get` or `/profile/me`

### Profile-Centric Auth Flow
- Better-Auth handles **authentication** (user identity, session)
- Custom treaty API handles **authorization** (profile, roles, permissions)
- User signs up via Better-Auth → gets authenticated session
- Separate profile creation grants app access + RBAC roles
- This separation allows auth-optional flows (e.g., browsing comics without account)

### Protected Route Access
1. **Frontend**: Request → Next.js API route `/api/...` (proxy)
2. **Proxy**: Forwards to `BACKEND_TARGET_URL` + Elysia route (server-side)
3. **Middleware**: Route uses `authenticationMiddleware.userAuth()` or `roleAuth()`
   - `userAuth`: Validates session via `auth.api.getSession({ headers })`
   - Optionally checks `profile` exists if `requiredProfile: true`
   - `roleAuth`: Validates session + profile + permission filter
     - Resolves permissions via `resolveUserPermissions(profile.rolesIDs)`
     - Evaluates `PermissionFilter` (string, and, or, not)
4. **Backend**: Returns context to handler OR 401/403 error

### Sign Out Flow
1. **Frontend**: `signOutMutation` → `authClient.signOut()` (Better-Auth)
2. **Better-Auth**: Invalidates session in DB, clears session cookie
3. **Frontend**: `queryClient.removeQueries({ queryKey: ["session"] })`
4. **Result**: Client is fully unauthenticated (no session, no profile)

### Session Refresh Flow
1. **Frontend**: `sessionQuery` auto-fetches on mount
2. **Better-Auth**: `authClient.getSession()` reads cookie, validates session
3. **Better-Auth**: Returns current session state (or null if expired)
4. **Frontend**: Query result provides `session` data for UI
- Better-Auth handles JWT refresh via cookie mechanism internally

## Security Model

- **Session tokens** stored in HTTP-only, secure cookies (prod)
- **Cross-subdomain cookies** enabled for SSO across subdomains
- **Backend host never exposed** to browser (Next.js proxy)
- **RBAC** via roles → permissions → route-level middleware
- **Profile required for app access** (configurable via `requiredProfile`)
- **Admin flag** (`createNewAdmin` app state) grants first user admin role
